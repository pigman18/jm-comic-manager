# 针对 AI 代理的行为守则

## 图片 / 视觉反馈
- **禁止** 在看到 "Cannot read image.png" 错误后继续盲猜——立即告诉用户我看不到图，请用文字描述问题
- **禁止** 在需要判断布局、间距、对齐等视觉效果时不确认就动手改——自己无法验证时先问清楚

## 样式修改
- **禁止** 每次只按用户说的一个点改，不改其他明显问题——改之前先通盘审视整体布局的合理性
- **禁止** 在用户否定方案后沿着同一个思路再试一次——说明方向错了，必须换思路
- **禁止** 给输入框/选择框随意设不统一的宽度——同一行的同类控件宽度必须一致
- **禁止** 用 `margin-left: auto` 把按钮推到行尾——这会制造大段空白，改用 `justify-content` 对齐整组
- **禁止** 把复选框或附属控件单独放一整行——占用高度，应该和其他控件同行排列
- **禁止** 在已有类似控件/样式的情况下无视旧格式写新格式——先看现有代码的命名、类名、结构，保持一致
- **禁止** 在没有明确要求隐藏控件的情况下，自作主张把控件隐藏或移除——始终保持控件可见，除非用户明确说"去掉/隐藏"

## 间距与排列
- **禁止** 移动元素位置后不管间距——把标签从英雄区外挪到 info 区内后，必须检查该元素的 margin 是否破坏父级 `gap` 的统一性，多余的 margin 必须清零

## 逻辑与组件复用
- **禁止** 弹窗和独立页面写两套实现——弹窗必须内嵌页面组件（Dialog 套 Page），样式只维护一份
- **禁止** `dialog` 模式下擅自隐藏页面内容区——`dialog` prop 仅控制：隐藏页面自己的头部/导航；内容区必须和页面模式完全一致
- **禁止** 不查现有逻辑就凭空写行为——标签/作者点击必须查原有 DetailPage 的 `filterByTag`/`filterByAuthor` 实现（跳 catalog 页），而不是自己发明跳 search 页
- **禁止** 用自己的 class 名覆盖全局已有的样式——先查 `app.css` 和现有组件有无 `jmz-chip`、`jmz-author-link` 等全局类，直接用，不要发明新的

## 整体设计
- **禁止** 只会从上往下堆——思考左右、内嵌、卡片等不同的布局方案
- **禁止** 不理解用户意图就动手——先确认"左搜索右拉取"到底是指什么
- **禁止** 等到用户指出每个细节才改——对于一个改动，要把相关的间距、对齐、尺寸一并调好
- **禁止** 弹窗和独立页面写两套实现——弹窗必须内嵌页面组件（Dialog 套 Page），样式只维护一份

## 文件操作安全
- **禁止** 使用 PowerShell 的 `Set-Content`/`Out-File`/`Add-Content` 修改含中文的文件——这些 cmdlet 默认使用系统 ANSI 编码，会破坏 UTF-8 编码的非 ASCII 字符。始终使用 `edit` 工具修改文件
- **禁止** 在确认文件编码无损前不做验证——每次文件写入后都应该检查 git diff 确认中文等非 ASCII 字符未被破坏，然后构建验证
- **禁止** 用 `git checkout` 回退已修改文件来修复编码损坏——这会撤销所有未提交的改动（包括正确的改动），应该在用 `edit` 工具前先 `git diff` 确认回退范围

## 加载状态与动画
- **禁止** 为同一加载状态叠加多个视觉指示——有转圈就不要同时显示"正在加载"文字，反之亦然，一个 loading 状态只能有一个视觉反馈
- **禁止** 修改 `n-spin`/`n-loading` 等包装型组件的显示条件后不验证内部是否有内容撑起遮罩层——空内容时遮罩层无高度，转圈不可见

## 改动范围控制
- **禁止** 一个 CSS 属性值调整带出多余属性——只改需要改的属性，不添加未经验证的无关声明（如给 sticky 元素加 `left:0;right:0`）
- **禁止** 为清理一行代码使用批处理脚本操作多个文件——始终用 `edit` 工具逐文件精确修改，不要用 `-replace` 做批量正则替换
- **禁止** 只改条件不改后果——`n-spin :show` 从 `loading && !comic` 变成 `loading` 后，必须验证 `n-spin` 包装型组件在各种状态下（空内容时遮罩层是否有高度）的行为；任何条件/属性的变更都需要通读目标组件的全部渲染分支和第三方库的行为约束

## Naive UI 样式覆盖
- **禁止** 用 CSS class + `!important` 和 Naive UI 的组件样式打架——Naive UI 内部有复杂的 hover/active 状态管理，用 `!important` 会导致鼠标移开才生效等诡异行为。需要让 Naive UI 组件的某个子元素呈现不同颜色时，直接通过子元素的内联 style 或组件 prop（如 `n-icon` 的 `color` prop）控制，不要试图在父级用 `!important` 覆盖
