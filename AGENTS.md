# 针对 AI 代理的行为守则

## 思维与流程（禁止跳过）
- **必须** 在修改任何交互逻辑前，先通读该页面的完整模板（找所有 `@click`/`@keyup.enter`/`@change`/`v-model`）和全部相关函数定义，知道谁绑了什么、谁调了什么，然后再动手。不读代码改出 bug 是我的核心死穴，读代码是唯一解药
- **必须** 在一处改动完成后，验证所有与之相关的交互路径——如果改了搜索按钮，必须点搜索、点页码、改筛选条件回车、浏览器前进后退，全走一遍
- **必须** 每次验证任何改动时，在浏览器里做以下全景测试（无论看起来是否相关）：
  a. 点击该页面的每个按钮
  b. 在每个输入框打字回车
  c. 点击每个页码
  d. 切换浏览器的前进/后退
  e. 切换到其他 tab 再切回来（keep-alive 恢复）
- **必须** 当用户说某个行为是错的时，立即停手，先读完整代码（模板绑定 + 所有相关函数定义），说出问题出在哪里、应该怎么改，让用户确认后再动手
- **必须** 当连续 2 次改动都在改同一个方向（比如连续换参数、连续调同一个 watch）时，立刻停下来——这说明方向错了。必须删除所有改动的痕迹（git checkout 恢复），重新通读代码，换一个思路
- **必须** 在每一轮修复中只改一个 bug。如果发现另一个不相关的问题，记录到 AGENTS.md 待办但不改，等用户自己提
- **禁止** 在改交互 bug 时引入 `watch`/`nextTick`/`cachedQueryKey`/abortController 等联动机制。95% 的交互 bug 是模板没绑函数或绑错了函数，不是缺联动逻辑
- **禁止** 在自己无法验证的场景（如视觉效果）下不确认就动手——自己看不到图、判断不了间距对齐时，必须问用户要文字描述，不要盲猜一个值就改
- **禁止** 把"用户报告 bug"当作"可以改其他东西"的许可——每行改动都必须直接关联到该 bug 的修复上，多改一行都不行

## 图片 / 视觉反馈
- **禁止** 在看到 "Cannot read image.png" 错误后继续盲猜——立即告诉用户我看不到图，请用文字描述问题
- **禁止** 在需要判断布局、间距、对齐等视觉效果时不确认就动手改——自己无法验证时先问清楚

## 样式修改
- **禁止** 每次只按用户说的一个点改，不改其他明显问题——改之前先通盘审视整体布局的合理性
- **禁止** 在用户否定方案后沿着同一个思路再试一次——说明方向错了，必须换思路
- **禁止** 给输入框/选择框随意设不统一的宽度——同一行的同类控件宽度必须一致
- **禁止** 用 `margin-left: auto` 把按钮推到行尾——这会制造大段空白，改用 `justify-content` 对齐整组
- **禁止** 把复选框或附属控件单独放一整行——占用高度，应该和其他控件同行排列
- **禁止** 在已有类似控件/样式的情况下无视旧格式写新格式——先看现有代码的命名、类名、结构，保持一致
- **禁止** 在没有明确要求隐藏控件的情况下，自作主张把控件隐藏或移除——始终保持控件可见，除非用户明确说"去掉/隐藏"
- **禁止** 等用户指出每个细节才改——对于一个改动，要把相关的间距、对齐、尺寸一并调好

## 间距与排列
- **禁止** 移动元素位置后不管间距——把标签从英雄区外挪到 info 区内后，必须检查该元素的 margin 是否破坏父级 `gap` 的统一性，多余的 margin 必须清零

## 逻辑与组件复用
- **禁止** 弹窗和独立页面写两套实现——弹窗必须内嵌页面组件（Dialog 套 Page），样式只维护一份
- **禁止** `dialog` 模式下擅自隐藏页面内容区——`dialog` prop 仅控制：隐藏页面自己的头部/导航；内容区必须和页面模式完全一致
- **禁止** 不查现有逻辑就凭空写行为——标签/作者点击必须查原有 DetailPage 的 `filterByTag`/`filterByAuthor` 实现（跳 catalog 页），而不是自己发明跳 search 页
- **禁止** 用自己的 class 名覆盖全局已有的样式——先查 `app.css` 和现有组件有无 `jmz-chip`、`jmz-author-link` 等全局类，直接用，不要发明新的

## 整体设计
- **禁止** 只会从上往下堆——思考左右、内嵌、卡片等不同的布局方案
- **禁止** 不理解用户意图就动手——先确认"左搜索右拉取"到底是指什么
- **禁止** 等到用户指出每个细节才改——对于一个改动，要把相关的间距、对齐、尺寸一并调好
- **禁止** 弹窗和独立页面写两套实现——弹窗必须内嵌页面组件（Dialog 套 Page），样式只维护一份

## 文件操作安全
- **禁止** 使用 PowerShell 的 `Set-Content`/`Out-File`/`Add-Content` 修改含中文的文件——这些 cmdlet 默认使用系统 ANSI 编码，会破坏 UTF-8 编码的非 ASCII 字符。始终使用 `edit` 工具修改文件
- **禁止** 在确认文件编码无损前不做验证——每次文件写入后都应该检查 git diff 确认中文等非 ASCII 字符未被破坏，然后构建验证
- **禁止** 用 `git checkout` 回退已修改文件来修复编码损坏——这会撤销所有未提交的改动（包括正确的改动），应该在用 `edit` 工具前先 `git diff` 确认回退范围

## 加载状态与动画
- **禁止** 为同一加载状态叠加多个视觉指示——有转圈就不要同时显示"正在加载"文字，反之亦然，一个 loading 状态只能有一个视觉反馈
- **禁止** 修改 `n-spin`/`n-loading` 等包装型组件的显示条件后不验证内部是否有内容撑起遮罩层——空内容时遮罩层无高度，转圈不可见

## 改动范围控制
- **禁止** 一个 CSS 属性值调整带出多余属性——只改需要改的属性，不添加未经验证的无关声明（如给 sticky 元素加 `left:0;right:0`）
- **禁止** 为清理一行代码使用批处理脚本操作多个文件——始终用 `edit` 工具逐文件精确修改，不要用 `-replace` 做批量正则替换
- **禁止** 只改条件不改后果——`n-spin :show` 从 `loading && !comic` 变成 `loading` 后，必须验证 `n-spin` 包装型组件在各种状态下（空内容时遮罩层是否有高度）的行为；任何条件/属性的变更都需要通读目标组件的全部渲染分支和第三方库的行为约束
- **禁止** 在已有相关内容的文件中做局部修改前不通读该段落的完整上下文——比如在卡片底部追加元素，必须先看同一行的兄弟元素和相邻行的结构，确认布局方式后再动手，不能只盯着要改的那一行

## Naive UI 样式覆盖
- **禁止** 用 CSS class + `!important` 和 Naive UI 的组件样式打架——Naive UI 内部有复杂的 hover/active 状态管理，用 `!important` 会导致鼠标移开才生效等诡异行为。需要让 Naive UI 组件的某个子元素呈现不同颜色时，直接通过子元素的内联 style 或组件 prop（如 `n-icon` 的 `color` prop）控制，不要试图在父级用 `!important` 覆盖

## Naive UI 离散 API
- **禁止** 在包含 `n-message-provider`/`n-notification-provider`/`n-dialog-provider` 的根组件（如 `App.vue`）里调用 `useMessage`/`useNotification`/`useDialog`——这些 hook 需要 provider 在祖先节点，而根组件自己是 provider 的父级，inject 找不到。必须使用 `createDiscreteApi` 并传入完整的 `configProviderProps: { theme, themeOverrides, locale, dateLocale }`，否则离散 API 会使用默认浅色主题，和 app 的暗色主题不一致

## 弹窗/提示规范
- **禁止** 使用浏览器原生 `alert`/`confirm`/`prompt`——项目中已使用 Naive UI，有 `useMessage`/`useDialog`/`n-modal` 等组件可用。列表页等非根组件直接用 `useMessage()`/`useDialog()`；根组件用 `createDiscreteApi`。所有用户交互提示必须用 Naive UI 组件实现
- **禁止** 在弹窗内容中使用原生 HTML 标签（`<p>`、`<span>` 等）做文字展示——应该使用 Naive UI 的 `<n-text>` 等组件，以保持字体、颜色、间距与整体主题一致
- **禁止** 自行编造 API 行为描述——对于第三方接口（如 JM 主站），未验证的返回信息不要写在 UI 上。修改提交前必须查阅接口文档或实测确认行为

## CSS 布局
- **禁止** 在已有 `overflow-y: auto` + `max-height` 的简单布局上一层层堆 `flex: 1` 链条——n-spin 等第三方组件内部 DOM 结构不可控，flex 链条容易断裂。先尝试调整 `max-height`/`calc()` 值，不行再考虑 flex 改造
- **禁止** 连续 3 次尝试同一个方向失败后不换思路——改同一个方向的参数超过 3 次还没解决，说明方向错了，必须退回去从根上重新分析

## Vue 模板
- **禁止** `v-for` 和 `v-if` 分开放——`v-if` 在 `v-for` 外部取不到循环变量的值，必须用 `<template v-for>` 把需要循环的内容包在一起，条件放内部
- **禁止** 用普通空格（` `）做内联分隔符——HTML 会折叠连续空格，导致视觉上粘连。必须用 `&nbsp;`（`\u00a0`）确保间距可见

## 功能改动守则
- **禁止** 擅自给交互行为附加语义——用户说"搜索不反应"，问题只是"触发搜索后列表没刷新"，我不应该去假设"搜索应该回到第 1 页"。搜索按钮 == 用当前条件重新查询，页码不动。除非用户明确要求"搜索后重置到首页"，否则不得改动页码
- **禁止** 连续尝试同一错误方向超过 2 次——当用户明确指出某个行为是错的（如"搜索不应该重置页码"），必须立即停手，退回去重新理解问题，而不是继续在该方向上修修补补
- **禁止** 一次修多个不相关的东西——用户报告一个 bug，只修那一个 bug。不要顺手"改进"其他看起来不顺眼的部分。每一步改动都必须是用户要求的或修复该 bug 所必需的
- **禁止** 不先读代码就动手改——遇到交互 bug，先通读模板中该控件的所有绑定（`@click`/`@keyup.enter`/`v-model`）以及所有相关函数（`doSearch`/`resetPage`/`clearNumber` 等），搞清楚每个函数在做什么、被谁调用，再动代码
- **禁止** 把"没触发请求"等同于"需要加复杂逻辑"——按钮点了列表不刷新，最直接的做法是检查按钮绑定的函数有没有调用加载方法，而不是引入 `watch`/`nextTick`/`cachedQueryKey` 等联动机制
- **禁止** 在用户明确否定后沿相同思路再试——当用户说"你别这样"后，还继续在同一方向调整参数再试，等于无视用户反馈。必须换方案或先问清楚

## 加载状态与动画
- **禁止** 为同一加载状态叠加多个视觉指示——有转圈就不要同时显示"正在加载"文字，反之亦然，一个 loading 状态只能有一个视觉反馈
- **禁止** 静默请求不显示加载提示——任何网络请求都要有对应的加载状态反馈（转圈/进度条/骨架屏），禁止不加任何提示直接请求

## 今天 2026-06-22 惨痛教训：控制台控制 + 全流程验证 + 测试优先

### 问题一对应关系
- **用户说"debug=false 控制台还是会出现"** → 正确的验证方式：写一个 10 行独立测试脚本 `test-sw.mjs`，直接测 `GetConsoleWindow + ShowWindow(SW_HIDE)`，30 秒就知道答案。但实际做了：猜 ShowWindow 在 kernel32（已在 AGENTS.md 写过 ShowWindow 是 user32）、换 FreeConsole、换 GUI+AllocConsole、换 stdout hook、重建→跑→猜→重建→跑→猜，耗了 2 小时
- **结果**：用户回滚后自己在 1 分钟内改好了（从 user32 正确加载 ShowWindow），整个问题就是 DLL 名错了

### 所有问题的汇总

#### 1. 不写测试脚本，纯靠猜（最致命的问题）
当需要验证一个底层 API（ShowWindow/FreeConsole/WriteConsoleW）是否在 pkg 环境下正常工作时：
- ❌ 做法：直接改主代码 `app.js` → `npm run package`（3 分钟）→ 跑 exe → 看效果 → 猜
- ✅ 正确做法：在 `packages/desktop/` 下写 `test-xxx.mjs` 独立脚本，用 `ewvjs-cli package test-xxx.mjs -o test-xxx -t node22-win-x64` 快速打包（30 秒），跑完读结果，验证一个 API 调用不超过 1 分钟

**实际损失**：
- `ShowWindow` 从 user32 加载的问题：写了 test-sw.mjs 后 30 秒就定位到了，但在此之前猜了 10+ 轮，尝试了 FreeConsole、GUI+AllocConsole、stdout hook 等错误方向
- `WriteConsoleW` 是否在 pkg 工作的问题：写了 test-wcw.mjs 后 30 秒就确认 `alloc=true wcw=true`，但在此之前已经绕了一大圈
- `FreeConsole` 是否工作的问题：写了 test-fc.mjs 后 30 秒确认 `free=true`，但之前已经在 main 代码里反复改了好几次

#### 2. 直接覆盖用户配置文件（两次！）
第一次：`Set-Content ... config.json -Value '{}'` — 清空了全部配置，导致 bundle.start() 报 `path undefined`
第二次：`Set-Content ... config.json -Value '{"debug":true}'` — 再次覆盖，只保留一个 debug 字段
**后果**：用户的登录信息、token、爬虫配置全部被清空，需要手动恢复

#### 3. 错误方向持续不改
- 用户说"debug=false 控制台还是出现" — 问题就是 `ShowWindow` 没从正确 DLL 加载，应该在 2 轮内确定
- 但实际尝试了：FreeConsole → GUI+AllocConsole+WriteConsoleW → CONSOLE+FreeConsole → stdout hook → ... 至少 5 个方向
- 每个方向都改了 package.json（子系统来回变）、app.js、logger.js，导致改动量爆炸

#### 4. 加了大量临时 debug 日志，污染代码
`fs.writeFileSync('C:\\Users\\pigman\\AppData\\Local\\Temp\\logger-call.txt', ...)` 在多个文件写了几十条，用户看到后说了"你写的什么垃圾"

#### 5. 用户说问题在 A，跑去查 B
- 用户说"debug=false 控制台出现" → 跑去查 bundle.start() 为什么会抛异常 → 原因：config 被我自己清空了，和问题无关
- 用户说"server start 后日志不输出" → 跑去查 bundle 的 HTTP 请求日志走什么通道 → 原因：GUI 子系统的 stdout 根本就是断开的

#### 6. 用户已告知的信息不记住
- 很早就说过 `ShowWindow` 是 `user32.dll`，不是 `kernel32.dll` — 仍然尝试从 kernel32 加载
- 很早就说过 `PROCESS_ATTACH` 相关的问题，但调试时完全没参考

### 对应的禁止事项

- **禁止** 需要验证底层 API 或行为时直接改主代码 — 必须先在 `packages/desktop/` 写独立测试脚本（`test-xxx.mjs`），用 `ewvjs-cli package` 快速打包（30s），用文件输出结果验证，1 分钟内确认答案
- **禁止** 为改一个属性值直接覆盖整个用户配置文件（如 `config.json`） — 必须用 `Read` 读取、`JSON.parse` 解析、改单字段、`JSON.stringify` + `Write` 写回，或直接通过应用的正常 API 修改
- **禁止** 连续尝试同一错误方向超过 2 次 —— 必须停手，先写独立测试脚本确认基础假设（如 API 是否可用、参数是否正确），再考虑换方案
- **禁止** 在用户报告 bug A 时排查无关模块 B —— 必须先确认用户说的现象能不能复现、影响范围在哪里，不要跳到无关模块里去查
- **禁止** 在主代码或主目录下加临时 debug 日志文件 —— 验证用独立测试脚本，测试完就删；主代码只包含最终的生产逻辑
- **禁止** 无视用户明确告知过的信息（如 `ShowWindow` 在 `user32.dll`）—— 如果自己不确定，先问用户确认，不要盲猜

## 第三方 API 接入规范
- **禁止** 在未验证的情况下猜测第三方 API 的数据结构——必须先写测试脚本（`test/*.js`）真实调用接口，打印返回内容，确认字段名和类型后再写接入代码
- **必须** 在 `test/` 目录下编写测试脚本，使用 `node test/xxx.js` 独立运行验证接口连通性和数据格式，测试通过后才修改主代码
- **禁止** 直接在主代码中"试"API 格式——改代码→运行→看结果→再改 的循环效率低下，测试脚本可以在 30 秒内给出答案

## 2026-06-23 教训：调用未定义的对象

### 问题
写 `app.relaunch()` 但文件中根本没有 `app` 这个对象，文件顶部的 import 是 `import { create_window, expose, start } from 'ewvjs'`，没有 Electron `app`

### 对应的禁止事项

- **禁止** 凭记忆写代码 —— 调用任何变量/API/对象前，必须先通读文件顶部所有 `import`/`require` 语句，确认该对象在当前文件的 scope 中可用。读文件必须从第 1 行开始读，确认 import 之后再写调用代码

## 分页规则
- **必须** 主站 API 返回了 `total` 字段（表示总条数）的，使用传统分页（`n-pagination` 页码组件），不要用无限滚动
- **必须** 只有 API 不返回 `total`、以返回空数据或 `error` 字段表示结束的，才用无限滚动
- **必须** crawler 方法的 page 入参从 1 开始（符合常识），调用接口时做兼容转换（如 page-1），而不是要求调用者传 0-based 的 page
