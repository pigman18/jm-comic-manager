# 禁止规则（血泪教训）

## 网速问题类
- **禁止** 在网络不通时反复重试——5次内不成功立刻交给用户处理
- **禁止** 花超过 5 分钟研究怎么下载一个文件——直接把链接给用户
- **禁止** 在下载失败时尝试各种工具组合（curl、Invoke-WebRequest、代理、--resolve）——一律视为网络不可用

## pkg 打包类
- **禁止** 在 Windows 上通过 Node.js 的 `execSync`/`spawnSync` 调用 pkg——只要通过 `child_process` 加 `env` 就会产生 55MB 损坏 exe（Node v22 bug）
- **禁止** 让 pkg-fetch 从源码编译 Node（>90 分钟）——发现下载失败立刻手动缓存二进制
- **禁止** 使用 `preload-native.js` 替代 `rcedit` 或正确配置 pkg 的 `native` 选项
- **禁止** 忘记 `--fallback-to-source` 参数——否则 pkg 会静默跳过无法预加载的模块
- **禁止** 把 `HTTPS_PROXY` 环境变量传递给 pkg 进程——会破坏构建

## 沟通类
- **禁止** 花超过 10 分钟解决同一个问题而不告知用户
- **禁止** 出现闪退/中断后不解释原因就继续操作
- **禁止** 假设某个操作会在合理时间内完成而不设超时或反馈进度

## 流程类
- **禁止** 多次尝试后不切换策略——一条路走到黑是最大的浪费
- **禁止** 在不确定根因时做复杂的修改——先验证最简单的假设
- **禁止** 忽略用户已有的工具和网络环境——用户本机下载 5 分钟就能完成的事，不要替用户硬扛 4 小时

## 构建环境类
- **禁止** 在构建前不验证 pkg-fetch 缓存二进制存在且 SHA 匹配
- **禁止** 假设本地 `node.exe` 可以替代 pkg-fetch 的打了补丁的 Node 二进制
- **禁止** 在不知晓 `--compress GZip` 参数的前提下，以文件大小判断 exe 是否损坏——压缩后 60MB 是正确的，不压缩 153MB 也是正确的
- **禁止** 使用 `npm run build` 调用 pkg——必须直接用 `node_modules/.bin/pkg.cmd` 或 `npx --yes @yao-pkg/pkg`，因为 npm scripts 中的 `npx` 行为不稳定（曾产生错误大小的 exe）
- **禁止** 让 `tray-hook.exe` 仅作为 pkg `assets` 嵌入——pkg 的 assets 不负责提取文件到磁盘，`spawn()` 需要的二进制必须通过 `build.mjs prepare` 复制到 `src/` 目录下
- **禁止** 在构建前不清理 `dist/` 目录——旧的失败产物会影响新构建结果的判断
